1. How It Works — End to End
1
Upload & Map
Drop in a CSV/Excel export from any ITSM tool, match its columns to fields the app understands, choose a date format, and confirm — nothing is imported until you review the mapping.
↓
2
Automatic Sensitive-Data Scan
Every uploaded row is scanned for card numbers, IBANs, phone numbers and similar patterns as it comes in — see the Sensitive Data Report.
↓
3
Run a Clustering Engine (A and/or B)
Traditional ML and Generative AI each group similar tickets independently — results are kept separate per engine, so running one never overwrites the other.
↓
4
Review Problem Clusters
Each cluster carries a confidence score, a trend (rising/falling/stable), and — where the data supports it — a regional breakdown. High-confidence, recurring, rising clusters get flagged as Problem Candidates.
↓
5
Drill Down (optional)
Re-cluster just one cluster's own tickets into finer sub-groups when a group looks too broad to act on directly.
↓
6
Lookup Tools
Smart Search (semantic search), Smart Assign (queue suggestion), and Possible Duplicates all run on demand, independent of the main clustering runs. (The Sensitive Data Report is a review of step 2's automatic scan, not something you separately run.)
↓
7
Capture AI-Assisted Notes
Build a prompt from a cluster's data, paste it into your own Microsoft Copilot chat (nothing is sent by this app), then save the response back — as a one-line AI Summary, a trend explanation, or full resolution notes.

3. The Two Clustering Engines
Both are genuinely AI — Traditional ML (statistical/classical) and Generative AI (embedding-based) are both branches of the same AI family tree. The choice is about cost, infrastructure, and semantic depth, not "AI vs. no AI." Both run live and fully on this machine today — neither is waiting on external API access.
DimensionApproach A · Traditional MLApproach B · Generative AI
How it worksTF-IDF word-frequency vectors + HDBSCAN density clusteringLocal sentence-embedding model (all-MiniLM-L6-v2) + cosine-similarity clustering
Semantic understandingModerate — needs vocabulary/stem overlapStrong — understands paraphrasing and context
CostOne-time build cost, no per-call feesNo per-call fees either — runs a local model, not a hosted API
Data privacyFully on-premFully on-prem — a hosted Azure OpenAI endpoint remains an optional future upgrade, not a requirement
ExplainabilityTransparent, rule-based — easier to justify to an auditorLess transparent (embedding-space "black box")
Best fitFast, explainable first pass; good for audit-sensitive reviewPrimary recommendation for ticket text, since reporters reword the same issue differently — semantic matching catches what keyword overlap misses
Practical guidance: run Traditional ML first as a fast baseline, then Generative AI for the semantic pass. Both are safe to run on the same data — nothing is overwritten between engines, only within the same engine on a re-run.
4. Assumptions Made
  • Uploaded data has at least one genuinely free-text field (a title/description) — clustering quality depends directly on how descriptive that text is; a file of only short codes will cluster poorly.
  • Queue, offering, application, country, and similar fields are treated as optional. When your source data doesn't map to them, the features that depend on them (queue categorization, regional trend breakdown, app filters) simply have nothing to show — this isn't an error state.
  • A single date format applies consistently to every row within one uploaded file, chosen once during the mapping step.
  • At least 8 tickets are needed before any cluster can form. Small datasets, or a narrow "Run on" scope (a single upload, a single offering), may legitimately produce zero clusters.
  • The local sentence-embedding model is treated as a reasonable stand-in for a hosted Azure OpenAI embeddings endpoint — good for general English support-ticket text, but not fine-tuned to any organization's specific acronyms or internal system names.
  • Sensitive-data scanning (regex + Luhn/IBAN checksum + phone-number parsing) is assumed to catch the common, well-structured formats — card numbers, IBANs, phone numbers — not a certified DLP/PII product covering every possible sensitive-data shape.
  • A ticket pair is assumed to be a likely duplicate only when similarity is very high (≥92%) and close in time (7 days) and shares a reporter or application — high text similarity alone is treated as insufficient, since two distinct-but-related issues can still read as similar.
  • Queue-suggestion voting trusts whatever queue values already exist in your uploaded data as ground truth — if those source labels are inconsistent, suggestions inherit that inconsistency.
  • AI Summary, Trend Explanation, and Resolution capture all assume a human in the loop: this app builds the prompt, but a person pastes it into their own Copilot chat and pastes the answer back — there's no server-side call to any LLM anywhere in this build.
  • SQLite is assumed sufficient for this scale of use (single team, moderate ticket volumes); a production multi-user rollout would need PostgreSQL, per the original proposal.
5. Known Limitations
  • No server-side LLM generation anywhere. Smart Search and Smart Assign rank/vote over existing tickets (retrieval, not generation); AI Summary, Trend Explanation, and resolution notes all require a manual copy/paste into an external chat tool (e.g. Copilot) this app doesn't control.
  • Re-running an engine replaces its previous top-level results, not merges with them — by design, so a stale run never lingers, but it means any manual edits tied to the old clusters (like resolution notes) don't automatically carry over to the new ones if the cluster boundaries shift.
  • Resolution notes are tied to a specific cluster row. If you reset or re-run a pipeline and the cluster IDs change, previously saved notes stay on the old (now-gone) cluster rather than following the "same" issue to its new cluster — this is the clearest current gap between "notes exist" and "notes are durable."
  • Queue suggestions need enough labeled data to vote from. With very few or heavily skewed existing queue values, suggestions get thin and less trustworthy — treat low-agreement suggestions as a starting guess, not a verdict.
  • No integration back into an external ITSM tool. Promoting a Problem Candidate into an actual ServiceNow/Jira Problem record is a manual step outside this app today — nothing here creates or updates records elsewhere.
  • Everything is user-triggered. There's no background scheduler re-running pipelines or rebuilding indexes as new tickets arrive — you decide when to re-scan, re-cluster, or rebuild.
  • Clipboard-copy conveniences (e.g. "Copy Prompt for Copilot") depend on browser clipboard permissions — locked-down or automated browser environments may block them; paste manually if so.
  • No granular in-project permissions yet. Projects isolate data from each other, but anyone with access to a project can see and edit everything inside it — there's no per-user role restriction within a single project.
