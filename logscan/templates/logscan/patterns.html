Help — How pattern discovery works

Reads the last ~5MB of the file, normalizes each line (timestamps, IPs, UUIDs, and standalone numbers replaced with placeholder tokens), then clusters the normalized lines by shared vocabulary — TF-IDF + HDBSCAN, the same technique Traditional ML clustering uses for tickets. Unclustered covers lines that didn't share enough in common with anything else in the window to form a pattern.

Confidence here is size and density only (how big the cluster is relative to the largest one found, and how tightly HDBSCAN believes the lines belong together) — unlike ticket clusters, there's no trend/recency signal yet, since that needs reliable per-line timestamp parsing across arbitrary log formats.

The example line is never shown raw — it's redacted through the same detection used for the Sensitive Data / Log PII Alerts reports before it's ever saved, so an email, card number, etc. that happens to land in the chosen example shows up as [EMAIL]/[CARD]/etc. instead of the real value.

Re-running replaces the previous results for this file entirely — it isn't a running history, just the latest look at the most recent window.
