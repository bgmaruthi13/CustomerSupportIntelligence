Help — Trigger modes, and how each one actually runs

On-demand — click Scan Now. This launches manage.py run_log_scan as a detached background process; the page redirects to a status screen that polls progress, since a large file won't finish before the request would time out.

Scheduled — nothing inside this app schedules it. An external Windows Task Scheduler job (or cron on Linux) is expected to run manage.py run_scheduled_scans periodically, which scans every active scheduled source. See deploy/windows/README.md for registering it.

Continuous (tailing) — a separate, always-running process (manage.py tail_log_sources), deployed as its own Windows Service, watches active continuous sources and scans new bytes as they're appended — real-time, not a periodic batch.

Every mode resumes from Scanned so far rather than re-reading the whole file each time — a growing log only ever has its new bytes scanned.

Directory / NAS share sources scan every file matching File pattern under a folder (a mapped NAS share works the same as a local folder) — each file tracks its own resume point independently, so a scan only reads whatever's new in whichever files actually changed, plus any brand-new files that have appeared since the last pass. Files that don't match the pattern are never opened at all.

Find Patterns is different from Scan Now — instead of scanning for sensitive data, it clusters the last ~5MB of the file by shared wording (ignoring timestamps, IPs, and IDs) to surface recurring message types, the same idea as Problem Clusters for tickets. It runs in the request (no background job) and only looks at a recent slice, not the whole file — path/upload sources only for now.
