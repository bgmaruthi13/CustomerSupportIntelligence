Help — How is a cluster flagged as a "Problem Candidate"?

A cluster earns the Candidate ribbon only when all three of these hold:

  1. Recurring count is at or above this run's recurring threshold — enough tickets to be a real pattern, not a one-off.
  2. Confidence is at or above this run's confidence threshold — the cluster is a tight, reliable grouping (not a loose bucket of vaguely-related tickets).
  3. Trend is Rising or Stable — never Falling. A pattern that's already cooling off isn't proposed as a new Problem Record.

Where the thresholds come from

The thresholds aren't fixed numbers — each pipeline run derives its own bar from that run's own distribution of clusters, so "Candidate" always means "stands out from this run's peers," regardless of dataset size:

  • Recurring threshold = the higher of 5 or the 75th percentile of recurring counts across this run's clusters.
  • Confidence threshold = the higher of 65% or the 50th percentile of confidence scores across this run's clusters.

On a small run with too few clusters to take a meaningful percentile from, it falls back to the fixed 5 / 65% baseline.

How confidence itself is calculated

Confidence is a weighted blend of three signals, floored at 30% and capped at 97%:

confidence = 30 + 67 × (0.40 × size_score + 0.35 × density + 0.25 × recency)
  • size_score — this cluster's size ÷ the largest cluster's size in the run
  • density — average membership strength (how tightly HDBSCAN thinks these tickets belong together)
  • recency — fraction of the cluster's tickets created in the last 90 days

How trend is calculated

Ticket count in the earliest third of the cluster's date range is compared against the latest third (the middle third is excluded to sharpen the contrast): more than 25% higher in the recent window = Rising, more than 25% lower = Falling, otherwise Stable. Hover any trend badge to see the exact counts and dates behind it.

Worked example

Suppose a run produces clusters with recurring counts [3, 4, 6, 8, 12, 22] and confidence scores [40, 55, 62, 70, 78, 85].

Recurring threshold = max(5, 75th percentile of counts) = max(5, 12) = 12
Confidence threshold = max(65, 50th percentile of scores) = max(65, 70) = 70

A cluster with 18 recurring tickets, 82% confidence, Rising trend → 18 ≥ 12 ✓, 82% ≥ 70% ✓, Rising ✓ → flagged as a Problem Candidate.
A cluster with 14 recurring tickets, 72% confidence, Falling trend → clears both numeric bars, but Falling trend fails the third check → not flagged.
A cluster with 9 recurring tickets, 90% confidence, Rising trend → high confidence, but 9 < 12 fails the recurring-count bar → not flagged.

This status is recomputed fresh every time a pipeline runs — it isn't carried over from a previous run, so re-running with more data (or a narrower dataset scope) can change which clusters qualify.
