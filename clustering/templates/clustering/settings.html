Help — Clustering Settings

Why the defaults are different for each engine — this is the one thing worth understanding before touching any toggle. Traditional ML is a bag-of-words method (TF-IDF): it has no concept of grammar, so aggressive cleanup — lowercasing, stripping punctuation/numbers, removing stopwords, stemming inflected forms to a shared root — removes noise and tightens its vocabulary, which is why its defaults lean toward "clean everything up." Generative AI is the opposite: the embedding model is a grammar/meaning model, trained on natural, intact English sentences — the same cleanup that helps TF-IDF actively degrades its understanding, which is why its defaults leave lowercasing, punctuation, stopwords, and stemming all off. An "everything is off" panel on the Generative AI tab isn't unfinished — it's the deliberately correct choice.

Why Title + Description is the default for both engines — titles are often terse ("Batch job failed" tells you little on their own), while descriptions usually carry the actual detail. For Traditional ML, the cleanup pipeline (HTML/URL/email stripping, stopword removal) already handles the noise that made raw descriptions unreliable in the past, so the extra vocabulary is now a net win rather than a risk. For Generative AI, more natural context is close to a pure upside — the embedding model was built to understand full sentences, so there's no "noisy text" downside to worry about the way there is for TF-IDF.

Why Traditional ML also defaults to stemming and a minimum token length of 3 — stemming collapses "timeout"/"timed out"/"failing"/"failed" down to one shared token, which directly tightens an otherwise-fragmented TF-IDF vocabulary (the same fix this app's own troubleshooting notes flagged as the natural next step after stopword removal). A minimum token length of 3 drops single/double-character fragments that regularly survive punctuation/number stripping without ever carrying real signal. Neither applies to Generative AI — stemming would turn real words into non-words the embedding model was never trained on.

Source — which mapped ticket fields get concatenated (in the order checked) into the text that's actually clustered.

Cleanup — removes things that were never meaningful natural language (HTML markup, URLs, emails, boilerplate footers, ticket-ID noise). Generally safe to leave on for both engines.

Text shape — lowercasing and punctuation/number stripping. Helps Traditional ML's keyword matching; can hurt Generative AI's embedding quality, since the model was trained on natural, grammatically intact text.

Tokens — stopword removal (with your own add/exclude lists on top of the standard English list), a minimum token length, and stemming (collapses "failed"/"failing"/"fails" to one token).

Vectorizer tuning (Traditional ML only) — corpus-level statistics, not per-ticket cleanup: drop terms that are too rare (typos) or too common (appear in nearly every ticket) to help distinguish clusters.

Boilerplate / Ticket-ID patterns — one entry per line. A plain line is matched as a literal phrase (safe default). Prefix a line with regex: to match it as a regular expression instead, e.g. regex: INC\d{7}.

Preview — runs the form's current values (even if unsaved) against your most recently created ticket, so you can see the net effect of several toggles stacked together before committing to Save.

Confidence Scoring — how it works — every cluster's Confidence % blends three signals: size (how big this cluster is relative to the biggest one in the same run), density (how tightly HDBSCAN believes these tickets belong together), and recency (what fraction of the cluster's tickets are from the last N days). Whatever three importance numbers you enter, they're normalized to sum to 100% automatically — "50/30/20" and "5/3/2" produce the exact same weighting. Worked example at the default weights (40% size, 35% density, 25% recency, floor 30, cap 97): a cluster that's 50% the size of this run's biggest cluster, 80% membership density, and 60% of its tickets from the last 90 days scores 0.40×50 + 0.35×80 + 0.25×60 = 20 + 28 + 15 = 63 — that's 63% of the way from floor to cap, so 30 + 0.63 × (97 − 30) ≈ 72% confidence. Changing these weights only affects the next pipeline run, same as every other Clustering Settings change — and since the Problem Candidate bar is computed from this run's own confidence distribution, changing the weights can also change which clusters clear that bar.
