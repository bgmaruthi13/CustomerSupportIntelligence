Help — Running this pipeline

Choosing a "Run on" scope

Defaults to all of this project's tickets. Narrow it to a single upload if you want results from just one file, or to a specific Offering if you only want to look for patterns within one product/service line — useful when a project's tickets span several offerings that shouldn't really be clustered together.

Granularity

Broad favors fewer, larger clusters. Balanced (default) is the recommended setting for most datasets. Fine-grained pushes for more, smaller, more homogeneous clusters — try this if Balanced still feels too coarse for your data. This doesn't change the underlying algorithm (still HDBSCAN), just how it's tuned to select clusters.

Reading the results

  • Clusters Found — the count of real (non-noise) groups HDBSCAN detected. It picks this number itself based on density; you don't set a target cluster count.
  • Unclustered (Noise) — tickets that didn't fit any dense group. Some noise is normal (one-off tickets, typos-only titles); a very high percentage usually means the text is too sparse or too generic to vectorize well.
  • Candidate ribbon — see the Help section on the Problem Clusters page for exactly how this is decided, with a worked example.

Re-running

Running again replaces this engine's previous results — it doesn't add to them. Reset Results clears them without a re-run, if you just want a clean slate; your tickets and uploads are never touched either way.
