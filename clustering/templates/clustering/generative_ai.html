Help — Running this pipeline

When to use this over Traditional ML

Use Generative AI when tickets describing the same issue are worded very differently ("batch job stuck" vs. "nightly process not finishing") — TF-IDF needs shared vocabulary to see the connection, embeddings capture the meaning instead. It's slower and CPU-heavier, so for a quick first pass on a large dataset, Traditional ML is usually the better starting point.

Choosing a "Run on" scope

Same options as Traditional ML — all tickets, one upload, or one Offering — useful for keeping semantically-similar-but-unrelated offerings from bleeding into each other's clusters.

Granularity

Same three presets as Traditional ML — Broad favors fewer, larger clusters, Balanced (default) is right for most datasets, Fine-grained pushes for more, smaller clusters if Balanced still feels too coarse.

Reading the results

Clusters Found, Unclustered (Noise), Confidence, and Trend all mean the same thing here as on Traditional ML — the scoring logic (clustering/scoring.py) is shared between both engines, only the vectorization differs. See the Problem Clusters page's Help section for the full Problem Candidate logic with a worked example.

Re-running

Running again replaces this engine's previous results. Reset Results clears them without a re-run. Neither touches Traditional ML's results, or your tickets/uploads.
